# 画面設計メモ

## 1. 目的

本ドキュメントは、「業務系SE向け AIテスト設計支援ツール」の Phase 1：ローカルMVP における画面設計を整理するものです。

Phase 1では、Next.js / React / TypeScript を使用し、ローカル環境で以下の操作ができる画面を作成します。

1. 仕様メモを入力する
2. テスト対象種別とテストレベルを選択する
3. LLM Mockでテスト設計結果を生成する
4. 生成結果を画面で確認する
5. Markdownをコピーする
6. 生成履歴を一覧で確認する
7. 生成履歴の詳細を確認する

MVPでは、高度なUI装飾よりも、業務アプリとして分かりやすく、ポートフォリオとして説明しやすい画面構成を優先します。

---

## 2. 画面設計方針

MVPにおける画面設計方針は以下です。

- 画面数は最小限にする
- 業務アプリとして分かりやすいレイアウトにする
- 入力、生成、結果確認、履歴確認の流れを明確にする
- セキュリティ注意書きを入力欄付近に表示する
- 架空サンプル仕様を使ってデモしやすくする
- 生成結果JSONとMarkdownの両方を確認できるようにする
- Markdownコピー機能を用意する
- 初期MVPでは認証、ユーザー管理、課金、ファイルアップロードは扱わない
- 初期MVPではレスポンシブ対応を作り込みすぎない
- READMEや面談で画面キャプチャを使って説明しやすい構成にする

---

## 3. 画面一覧

MVPで作成する画面は以下の3つです。

| 画面名 | パス | 目的 |
|---|---|---|
| テスト設計生成画面 | `/` | 仕様入力、生成実行、結果表示、Markdownコピー |
| 履歴一覧画面 | `/history` | 過去の生成履歴を一覧表示 |
| 履歴詳細画面 | `/history/[id]` | 過去の入力内容・生成結果・Markdownを表示 |

---

## 4. 画面遷移

画面遷移は以下です。

| 遷移元 | 操作 | 遷移先 |
|---|---|---|
| テスト設計生成画面 | 履歴一覧リンクをクリック | 履歴一覧画面 |
| 履歴一覧画面 | 詳細リンクをクリック | 履歴詳細画面 |
| 履歴一覧画面 | 新規生成リンクをクリック | テスト設計生成画面 |
| 履歴詳細画面 | 履歴一覧へ戻るリンクをクリック | 履歴一覧画面 |
| 履歴詳細画面 | 新規生成リンクをクリック | テスト設計生成画面 |

MVPでは、生成完了後に自動で履歴詳細へ遷移するのではなく、同じ画面内に生成結果を表示します。

理由：

- 入力から生成結果確認までの流れが分かりやすい
- Markdownコピーまでの操作が短い
- 実装が単純になる
- 画面キャプチャでMVPの価値を説明しやすい

## 6. テスト設計生成画面

### 6.1 基本情報

| 項目 | 内容 |
|---|---|
| 画面名 | テスト設計生成画面 |
| パス | `/` |
| 主な役割 | 仕様入力、生成実行、結果表示、Markdownコピー |

---

### 6.2 画面構成

テスト設計生成画面は、以下の構成にします。

1. 画面タイトル
2. セキュリティ注意書き
3. サンプル仕様選択エリア
4. 入力フォーム
5. 生成ボタン
6. エラー表示エリア
7. 生成結果表示エリア
8. Markdown表示・コピーエリア

---

### 6.3 画面タイトル

表示例：

```text
テスト設計生成
```

説明文：

```text
仕様メモを入力すると、LLM Mockがテスト観点・テストケース案・追加確認事項を生成します。
```

---

### 6.4 セキュリティ注意書き

入力フォームの上部に、以下の注意書きを常時表示します。

```text
注意：
顧客名、個人情報、APIキー、パスワード、本番環境情報、業務機密を含む情報は入力しないでください。
必要に応じて、架空データへの置き換えやマスキングを行ってから利用してください。
```

この注意書きは、非表示にしない方針とします。

理由：

- 業務系SE向けツールとして守秘義務を明確にするため
- ポートフォリオとしてセキュリティ意識を示すため
- 将来OpenAI APIやBedrock連携を行う場合の前提にもなるため

---

### 6.5 サンプル仕様選択エリア

MVPでは、架空のサンプル仕様を選択できるようにします。

表示例：

| サンプル名 | 対象種別 | 説明 |
|---|---|---|
| ログイン画面 | 画面 | 入力チェック、認証失敗、エラーメッセージ確認 |
| 注文登録API | API | リクエスト、レスポンス、DB更新、外部連携確認 |
| 日次売上集計バッチ | バッチ | 対象データ、異常終了、再実行、ログ確認 |
| 顧客マスタ更新処理 | DB更新 | 登録、更新、整合性、排他確認 |
| 在庫照会外部連携 | 外部連携 | タイムアウト、リトライ、異常応答確認 |

MVPでは、サンプル仕様はDB管理せず、Frontend側の定数として持ちます。

サンプル仕様を選択した場合、以下の入力項目に値を反映します。

- タイトル
- テスト対象種別
- テストレベル
- 仕様メモ
- 補足事項

---

### 6.6 入力フォーム項目

| 項目 | UI部品 | 必須 | 初期値 | 備考 |
|---|---|---:|---|---|
| タイトル | text input | 任意 | 空 | 未入力時はBackendで自動生成 |
| テスト対象種別 | select | 必須 | `api` | 画面/API/バッチ/DB更新/外部連携/その他 |
| テストレベル | select | 必須 | `integration` | 単体/結合/システム/受入 |
| 仕様メモ | textarea | 必須 | 空 | メイン入力 |
| 補足事項 | textarea | 任意 | 空 | 前提条件、制約、追加観点など |

---

### 6.7 テスト対象種別の選択肢

| 値 | 表示名 |
|---|---|
| `screen` | 画面 |
| `api` | API |
| `batch` | バッチ |
| `db` | DB更新 |
| `external` | 外部連携 |
| `other` | その他 |

---

### 6.8 テストレベルの選択肢

| 値 | 表示名 |
|---|---|
| `unit` | 単体テスト |
| `integration` | 結合テスト |
| `system` | システムテスト |
| `acceptance` | 受入テスト |

---

### 6.9 入力フォームのバリデーション

Frontendでは、最低限以下のバリデーションを行います。

| 項目 | バリデーション |
|---|---|
| テスト対象種別 | 未選択不可 |
| テストレベル | 未選択不可 |
| 仕様メモ | 未入力不可、空白のみ不可 |

タイトルは任意です。  
未入力の場合はBackend側で自動生成します。

補足事項は任意です。

---

### 6.10 生成ボタン

ボタン名：

```text
テスト設計を生成
```

動作：

1. 入力値を検証する
2. `POST /api/test-designs` を呼び出す
3. 生成中はボタンを無効化する
4. 成功時は生成結果を画面に表示する
5. 失敗時はエラーメッセージを表示する

生成中の表示例：

```text
生成中...
```

---

### 6.11 エラー表示

API呼び出し失敗時や入力不備がある場合、フォーム付近にエラーを表示します。

表示例：

```text
仕様メモを入力してください。
```

```text
テスト設計結果の生成に失敗しました。入力内容を確認して再実行してください。
```

MVPでは、エラー詳細を過度に細かく表示しません。

---

### 6.12 生成結果表示エリア

生成成功時に、以下を表示します。

1. 概要
2. テスト観点
3. テストケース案
4. 追加確認事項

#### 概要

表示項目：

- `summary`

#### テスト観点

表示項目：

- カテゴリ名
- 観点リスト

表示例：

```text
正常系
- 必須項目をすべて指定した場合に処理が成功すること
- 複数明細を含む注文情報が正しく処理されること
```

#### テストケース案

表示形式：

| No | タイトル | 前提条件 | 期待結果 | 優先度 |
|---|---|---|---|---|

手順は、MVPでは詳細表示または箇条書き表示とします。

#### 追加確認事項

表示例：

```text
- 在庫引当失敗時のエラーコード仕様を確認する必要があります。
- 注文登録後の通知処理有無を確認する必要があります。
```

---

### 6.13 Markdown表示・コピーエリア

生成結果の下に、Markdown表示エリアを用意します。

表示内容：

- `result_markdown`

操作：

- Markdownコピー

ボタン名：

```text
Markdownをコピー
```

コピー成功時の表示例：

```text
コピーしました
```

MVPでは、Markdownの編集機能は作成しません。  
Markdownは生成結果の表示・コピー用途に限定します。

|---|---|

| 主な役割 | 保存済みの生成履歴を一覧表示する |

### 7.2 画面構成

履歴一覧画面は、以下の構成にします。

1. 画面タイトル
2. 新規生成画面へのリンク
3. 履歴一覧テーブル
4. データなし表示
5. エラー表示

---

### 7.3 画面タイトル

表示例：

```text
生成履歴一覧
```

説明文：

```text
過去に生成したテスト設計結果を確認できます。
```

---

### 7.4 履歴一覧テーブル

表示項目：

| 項目 | 内容 |
|---|---|
| 作成日時 | `created_at` |
| タイトル | `title` |
| テスト対象種別 | `target_type` の表示名 |
| テストレベル | `test_level` の表示名 |

```text
作成日時の降順
```

```text
詳細を見る
```

履歴が0件の場合、以下を表示します。

```text

```

---

### 7.6 エラー表示

履歴一覧APIの取得に失敗した場合、以下を表示します。

```text
生成履歴の取得に失敗しました。時間をおいて再度確認してください。
```

---

### 7.7 MVPで作らない機能

履歴一覧画面では、MVP時点で以下は作成しません。

| 機能 | 理由 |
|---|---|
| 検索 | 初期履歴件数が少ない前提のため |
| フィルタ | MVPでは対象外 |
| ページング | 初期履歴件数が少ない前提のため |
| ソート切替 | 作成日時降順で十分なため |
| 履歴削除 | 初期MVPでは参照中心とするため |
| 履歴編集 | 生成履歴は編集しない方針のため |

## 10. 状態管理方針

MVPでは、複雑な状態管理ライブラリは使用しません。

| 対象 | 方針 |
|---|---|
| フォーム入力値 | Reactのローカルstateで管理 |
| 生成結果 | 生成画面のローカルstateで管理 |
| ローディング状態 | 画面単位のローカルstateで管理 |
| エラー表示 | 画面単位のローカルstateで管理 |
| 履歴一覧 | 画面表示時にAPI取得 |
| 履歴詳細 | 画面表示時にAPI取得 |

MVPでは、Redux、Zustand、React Queryなどは必須にしません。

理由：

- 実装範囲を抑えるため
- 初期MVPでは画面数が少ないため
- コードの説明を簡単にするため

## 11. UIコンポーネント方針

MVPでは、以下のような単純なコンポーネント分割を想定します。

| コンポーネント | 役割 |
|---|---|
| `Header` | アプリ名とナビゲーション |
| `SecurityNotice` | セキュリティ注意書き |
| `TestDesignForm` | 仕様入力フォーム |
| `SampleSpecSelector` | サンプル仕様選択 |
| `GenerationResult` | 生成結果表示 |
| `MarkdownPreview` | Markdown表示・コピー |

実装時は、過剰に細かく分割しすぎず、読みやすさを優先します。

## 13. サンプル仕様方針

MVPでは、架空のサンプル仕様を用意します。

サンプル仕様の目的は以下です。

- 守秘義務リスクを避ける
- READMEや面談でデモしやすくする
- 画面キャプチャを作りやすくする
- ユーザーが入力内容に迷わず動作確認できるようにする

MVPで用意するサンプル候補：

| サンプル | 対象種別 | テストレベル |
|---|---|---|
| ログイン画面 | `screen` | `acceptance` |
| 注文登録API | `api` | `integration` |
| 日次売上集計バッチ | `batch` | `system` |
| 顧客マスタ更新処理 | `db` | `integration` |
| 在庫照会外部連携 | `external` | `integration` |

---

## 14. セキュリティ・守秘義務上の注意

画面上では、入力欄の近くに以下の注意書きを表示します。

```text
注意：
顧客名、個人情報、APIキー、パスワード、本番環境情報、業務機密を含む情報は入力しないでください。
必要に応じて、架空データへの置き換えやマスキングを行ってから利用してください。
```

画面設計上の方針：

- 注意書きは初期表示時から見える位置に置く
- 入力フォームの下ではなく、入力前に目に入る位置に置く
- 閉じるボタンは設けない
- READMEやdocsにも同じ趣旨を記載する

---

## 15. MVPでは作らない画面

Phase 1の初期MVPでは、以下の画面は作成しません。

| 画面 | 作成しない理由 |
|---|---|
| ログイン画面 | 認証はMVP対象外のため |
| ユーザー管理画面 | マルチユーザー化はMVP対象外のため |
| 設定画面 | 初期MVPでは設定項目が少ないため |
| プロンプトテンプレート管理画面 | Phase 4以降の拡張候補のため |
| ファイルアップロード画面 | セキュリティリスクと実装範囲が増えるため |
| Excel出力画面 | Markdown出力後の拡張で十分なため |
| LLM Provider切替画面 | 初期MVPではMock固定のため |
| 利用履歴・料金表示画面 | 外部LLM APIを使わないため |

---

## 16. 画面テスト方針

MVPでは、最低限以下を確認します。

| 画面 | 確認内容 |
|---|---|
| テスト設計生成画面 | 入力フォームが表示される |
| テスト設計生成画面 | セキュリティ注意書きが表示される |
| テスト設計生成画面 | 必須項目未入力時にエラーが出る |
| テスト設計生成画面 | 生成APIを呼び出せる |
| テスト設計生成画面 | 生成結果が表示される |
| テスト設計生成画面 | Markdownをコピーできる |
| 履歴一覧画面 | 履歴一覧が表示される |
| 履歴一覧画面 | 履歴が0件の場合の表示が出る |
| 履歴詳細画面 | 入力内容と生成結果が表示される |
| 履歴詳細画面 | Markdownをコピーできる |
| 共通 | 主要画面へ遷移できる |

MVPでは、E2Eテストを必須にはしません。  
まずは手動確認と、必要に応じた最小限のコンポーネントテストで十分です。

## 17. 画面キャプチャ方針

画面キャプチャには、必ず架空データを使用します。
