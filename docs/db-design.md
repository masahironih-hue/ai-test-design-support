# DB設計メモ

## 1. 目的

本ドキュメントは、「業務系SE向け AIテスト設計支援ツール」の Phase 1：ローカルMVP におけるDB設計を整理するものです。

Phase 1では、SQLiteを使用して、テスト設計生成履歴をローカルに保存します。

MVPの目的は、複雑なDB設計を行うことではなく、以下の流れを実現するための最小限の永続化を行うことです。

```text
仕様入力
↓
LLM Mockによる生成
↓
生成結果表示
↓
Markdownコピー
↓
履歴保存
↓
履歴一覧・履歴詳細確認
```

## 更新メモ

本ドキュメントは、ローカルMVP総合動作確認後の実装状態に合わせて、APIパス、入力項目名、`target_type` / `test_level` のコード値を更新しています。

- 生成API: `POST /test-designs/generate`
- 履歴一覧API: `GET /test-designs/histories`
- 履歴詳細API: `GET /test-designs/histories/{history_id}`
- 入力項目: `title`、`target_type`、`test_level`、`spec_text`、`supplement`
- `target_type`: `screen`、`api`、`batch`、`db`、`external`
- `test_level`: `unit`、`integration`、`system`

---

## 2. DB設計方針

MVPにおけるDB設計方針は以下です。

- SQLiteを使用する
- 履歴保存用テーブルは1つにする
- 入力内容、生成結果JSON、Markdownを1レコードにまとめて保存する
- 初期MVPでは正規化しすぎない
- テスト観点やテストケースは別テーブルに分けない
- 履歴の編集・削除・検索はMVPでは対象外とする
- 外部LLM API利用履歴はMVPでは詳細管理しない
- 本業の顧客情報、個人情報、APIキー、パスワード、業務機密は保存しない
- SQLiteのDBファイルはGit管理対象外とする

---

## 3. 使用DB

MVPでは、以下を使用します。

| 項目 | 内容 |
|---|---|
| DB | SQLite |
| 用途 | ローカルMVPの履歴保存 |
| ORM候補 | SQLModel |
| マイグレーション | MVP初期では必須にしない |
| DBファイル | `backend/app.db` または `backend/data/app.db` |

ORMは、FastAPIとの相性と実装の分かりやすさを優先し、MVPでは `SQLModel` を優先候補とします。

ただし、実装時にSQLAlchemyの方が扱いやすい場合は、SQLAlchemyへ変更してもよいものとします。

1回の生成処理に対して、1レコードを保存します。

| カラム名 | 型 | 必須 | 主キー | 内容 |
|---|---|---:|---:|---|
| `id` | integer | 必須 | 〇 | 履歴ID |
| `title` | varchar | 必須 |  | テスト設計タイトル |
| `target_type` | varchar | 必須 |  | テスト対象種別 |
| `test_level` | varchar | 必須 |  | テストレベル |
| `spec_text` | text | 必須 |  | 入力された仕様メモ |
| `supplement` | text | 任意 |  | 補足事項 |
| `result_json` | text | 必須 |  | 生成結果JSON文字列 |
| `result_markdown` | text | 必須 |  | Markdown形式の生成結果 |
| `llm_provider` | varchar | 必須 |  | LLM Provider |
| `llm_model` | varchar | 必須 |  | LLM Model |
| `created_at` | datetime | 必須 |  | 作成日時 |
| `updated_at` | datetime | 必須 |  | 更新日時 |

## 6. カラム詳細

### 6.1 `id`

履歴を一意に識別するIDです。

| 項目 | 内容 |
|---|---|
| 型 | integer |
| 必須 | 必須 |
| 主キー | 〇 |
| 採番 | 自動採番 |

---

### 6.2 `title`

テスト設計のタイトルです。

| 項目 | 内容 |
|---|---|
| 型 | varchar |
| 必須 | 必須 |
| 例 | `注文登録APIのテスト設計` |

ユーザーがタイトルを未入力にした場合は、Backend側で以下のようなタイトルを自動生成します。

```text
テスト設計 2026-05-21 10:00
```

---

### 6.3 `target_type`

テスト対象種別です。

| 項目 | 内容 |
|---|---|
| 型 | varchar |
| 必須 | 必須 |

想定値は以下です。

| 値 | 表示名 | 内容 |
|---|---|---|
| `screen` | 画面 | 画面仕様、入力チェック、表示制御など |
| `api` | API | REST API、内部API、外部公開APIなど |
| `batch` | バッチ | 日次処理、集計処理、ファイル取込など |
| `db` | DB更新 | 登録、更新、削除、整合性確認など |
| `external` | 外部連携 | 外部API、ファイル連携、タイムアウト、リトライなど |

MVPでは、DB制約としてCHECK制約を厳密に設定するかどうかは必須にしません。  
まずはBackend側のバリデーションで制御します。

---

### 6.4 `test_level`

テストレベルです。

| 項目 | 内容 |
|---|---|
| 型 | varchar |
| 必須 | 必須 |

想定値は以下です。

| 値 | 表示名 | 内容 |
|---|---|---|
| `unit` | 単体テスト | 関数、メソッド、単一処理単位の確認 |
| `integration` | 結合テスト | API、DB、外部連携を含む結合確認 |
| `system` | システムテスト | システム全体としての業務シナリオ確認 |

MVPでは、Backend側のバリデーションで制御します。

---

### 6.5 `spec_text`

ユーザーが入力した仕様メモです。

| 項目 | 内容 |
|---|---|
| 型 | text |
| 必須 | 必須 |
| 例 | `注文情報を受け取り、注文テーブルと注文明細テーブルに登録するAPI。` |

この項目には、架空データまたはマスキング済みデータのみを保存します。

以下の情報は保存しません。

- 顧客名
- 個人情報
- APIキー
- パスワード
- アクセストークン
- 本番環境情報
- 実際の設計書
- 実際のソースコード
- 実際の障害ログ
- 業務機密
- 契約上公開できない情報

---

### 6.6 `supplement`

補足事項です。

| 項目 | 内容 |
|---|---|
| 型 | text |
| 必須 | 任意 |
| 例 | `在庫引当失敗時は注文登録を行わず、エラーレスポンスを返す。` |

未入力の場合は、空文字またはNULLを許容します。  
実装時は、扱いやすさを優先して空文字に統一してもよいものとします。

---

### 6.7 `result_json`

LLM Mockによる生成結果JSONを文字列として保存します。

| 項目 | 内容 |
|---|---|
| 型 | text |
| 必須 | 必須 |
| 保存形式 | JSON文字列 |

保存例：

```json
{
  "summary": "注文登録APIに対するテスト観点とテストケース案です。",
  "viewpoints": [
    {
      "category": "正常系",
      "items": [
        "必須項目をすべて指定した場合に処理が成功すること"
      ]
    }
  ],
  "test_cases": [
    {
      "case_no": "TC-001",
      "title": "正常な注文登録",
      "precondition": "有効な顧客情報と商品情報が存在する",
      "steps": [
        "正常な注文情報を送信する",
        "レスポンス内容を確認する"
      ],
      "expected_result": "注文登録が成功する",
      "priority": "High"
    }
  ],
  "additional_questions": [
    "在庫引当失敗時のエラーコード仕様を確認する必要があります。"
  ]
}
```

MVPでは、JSON構造をDB上で分解せず、1つの文字列として保存します。

理由：

- 実装が単純になる
- 入力と出力の対応関係を保持しやすい
- Frontendにそのまま返しやすい
- 将来必要になった時点で正規化やJSON検索を検討できる

---

### 6.8 `result_markdown`

生成結果をMarkdown形式に変換した文字列です。

| 項目 | 内容 |
|---|---|
| 型 | text |
| 必須 | 必須 |
| 用途 | 画面表示、コピー、履歴詳細表示 |

保存例：

```markdown
# テスト設計結果

## 入力情報

- タイトル: 注文登録APIのテスト設計
- テスト対象種別: API
- テストレベル: 結合テスト

## 概要

注文登録APIに対するテスト観点とテストケース案です。
```

Markdownは、生成時点の内容をそのまま保存します。  
履歴詳細画面では、再生成せず、保存済みMarkdownを表示します。

| 項目 | 内容 |

### 6.11 `created_at`

レコード作成日時です。

| 項目 | 内容 |
|---|---|
| 型 | datetime |
| 必須 | 必須 |
| 用途 | 履歴一覧の並び順、作成日時表示 |

MVPではローカル開発環境での利用を前提とし、単純な日時として扱います。

---

### 6.12 `updated_at`

レコード更新日時です。

| 項目 | 内容 |
|---|---|
| 型 | datetime |
| 必須 | 必須 |
| 用途 | 将来の履歴編集・再生成対応に備える |

MVPでは履歴編集を行わないため、作成時点では `created_at` と同じ値になります。

ただし、将来の履歴更新、再生成、メモ追加などに備えてカラムを用意します。

## 7. インデックス方針

MVPでは、明示的なインデックスは最小限とします。

| 対象 | 方針 |
|---|---|
| `id` | 主キーのため自動的に利用 |
| `created_at` | 履歴一覧を作成日時降順で取得するため、必要に応じて追加 |
| `target_type` | MVPでは検索しないため不要 |
| `test_level` | MVPでは検索しないため不要 |

MVP初期では、履歴件数が少ない前提のため、`created_at` インデックスは必須ではありません。

ただし、履歴一覧取得で以下のような並び順を使います。

```sql
ORDER BY created_at DESC
```

履歴件数が増えた場合は、`created_at` へのインデックス追加を検討します。

---

## 8. データ保存タイミング

履歴データは、テスト設計生成APIの実行時に保存します。

対象API：

```text
POST /test-designs/generate
```

保存タイミング：

```text
入力値バリデーション
↓
LLM Mock生成
↓
Markdown生成
↓
DB保存
↓
保存結果をレスポンス返却
```

MVPでは、生成のみ実行して保存しないモードは作成しません。

## 10. DBファイルのGit管理方針

SQLiteのDBファイルはGit管理対象外とします。

理由：

- ローカル実行時に生成されるファイルであるため
- 履歴データに入力内容が含まれるため
- 誤って機密情報を含むデータをコミットするリスクを避けるため
- 環境ごとにDBを作成すればよいため

`.gitignore` に以下を追加します。

```gitignore
# SQLite database files
*.db
*.sqlite
*.sqlite3
backend/app.db
backend/data/*.db
```

DBファイルそのものではなく、テーブル定義や起動手順をdocsに残します。

---

## 11. 初期データ方針

MVPでは、DBに初期データを投入することは必須にしません。

サンプル仕様は、DBではなくFrontend側のサンプル入力として保持します。

理由：

- DB設計を単純に保つため
- サンプル仕様と履歴データの責務を分けるため
- 履歴テーブルは実際に生成した結果のみを保存するため

将来的にサンプル仕様をDB管理したくなった場合は、別途サンプル仕様テーブルを検討します。

---

## 12. MVPでは作成しないテーブル

Phase 1の初期MVPでは、以下のテーブルは作成しません。

| テーブル | 作成しない理由 |
|---|---|
| `users` | 認証・ユーザー管理はMVP対象外のため |
| `test_design_viewpoints` | 生成結果はJSON文字列で保存すれば十分なため |
| `test_cases` | 初期MVPではテストケースを正規化しないため |
| `sample_specs` | サンプル仕様はFrontend側で保持するため |
| `llm_usage_logs` | 外部LLM APIを使わないため |
| `prompt_templates` | プロンプトテンプレート管理はMVP対象外のため |
| `uploaded_files` | ファイルアップロードはMVP対象外のため |

## 13. 将来の拡張候補

MVP完了後、必要に応じて以下の拡張を検討します。

| 拡張 | 内容 |
|---|---|
| 履歴削除 | 不要な履歴を削除できるようにする |
| 履歴検索 | タイトル、対象種別、テストレベルで検索できるようにする |
| ページング | 履歴件数が増えた場合に対応する |
| 生成結果の正規化 | 観点・テストケースを別テーブル化する |
| OpenAI利用履歴 | LLM Provider、Model、Token数、料金目安を保存する |
| Bedrock利用履歴 | Bedrock利用時のモデル情報や利用回数を保存する |
| PostgreSQL対応 | AWS実務アピール版でRDS等を検討する場合に対応する |
| Alembic導入 | テーブル変更履歴を管理する |
| サンプル仕様管理 | サンプル仕様をDBで管理する |
| Markdown再生成 | 保存済みJSONからMarkdownを再生成する |

## 14. セキュリティ・守秘義務上の注意

本DBには、入力された仕様メモと生成結果が保存されます。

そのため、以下の情報は絶対に保存しません。

- 顧客名
- 個人情報
- APIキー
- パスワード
- アクセストークン
- 本番環境情報
- 実際の設計書
- 実際のソースコード
- 実際の障害ログ
- 業務機密
- 契約上公開できない情報

動作確認には、必ず架空データまたはマスキング済みデータを使用します。

また、SQLiteのDBファイルはGit管理対象外とし、誤ってコミットしないようにします。

---

## 15. 完了条件

DB設計・実装の完了条件は以下です。

- SQLiteを利用する方針が決まっている
- `test_design_histories` テーブルが作成されている
- 入力内容を保存できる
- 生成結果JSONを保存できる
- Markdownを保存できる
- 履歴一覧用のデータを取得できる
- 履歴詳細用のデータを取得できる
- SQLiteのDBファイルがGit管理対象外になっている
- 実データや機密情報を使わず、架空データで動作確認できる
