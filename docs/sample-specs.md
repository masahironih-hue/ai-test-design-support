# サンプル仕様メモ

## 1. 目的

本ドキュメントは、「業務系SE向け AIテスト設計支援ツール」の Phase 1：ローカルMVP で使用する架空サンプル仕様を整理するものです。

サンプル仕様は、以下の用途で使用します。

- ローカルMVPの動作確認
- 生成画面のサンプル入力
- README用の画面キャプチャ
- 面談時のデモ説明
- テスト設計支援ツールとしてのユースケース説明

本ドキュメントに記載する仕様はすべて架空データです。

---

## 2. サンプル仕様方針

MVPで使用するサンプル仕様の方針は以下です。

- 本業の顧客情報は使用しない
- 実在するシステム名、会社名、個人名は使用しない
- 実際の設計書、実コード、実ログは使用しない
- APIキー、パスワード、アクセストークンは記載しない
- 業務機密や契約上公開できない情報は含めない
- 業務系システムでよくある題材にする
- 画面、API、バッチ、DB更新、外部連携を幅広く含める
- テスト観点が出しやすい程度の具体性を持たせる
- MVPのデモに使いやすい長さにする

---

## 3. サンプル仕様一覧

MVPでは、以下の5件をサンプル仕様として用意します。

| No | サンプル名 | 対象種別 | テストレベル | 主な観点 |
|---:|---|---|---|---|
| 1 | ログイン画面 | 画面 | 受入テスト | 入力チェック、認証失敗、エラーメッセージ |
| 2 | 注文登録API | API | 結合テスト | リクエスト、レスポンス、DB更新、外部連携 |
| 3 | 日次売上集計バッチ | バッチ | システムテスト | 対象データ、異常終了、再実行、ログ |
| 4 | 顧客マスタ更新処理 | DB更新 | 結合テスト | 登録、更新、整合性、排他、トランザクション |
| 5 | 在庫照会外部連携 | 外部連携 | 結合テスト | タイムアウト、リトライ、異常応答、ログ |

---

## 4. サンプル1：ログイン画面

### 4.1 基本情報

| 項目 | 内容 |
|---|---|
| タイトル | ログイン画面のテスト設計 |
| 対象種別 | `screen` |
| テストレベル | `acceptance` |

### 4.2 仕様メモ

ユーザーIDとパスワードを入力してログインする画面。

ユーザーIDとパスワードが正しい場合は、トップ画面へ遷移する。

ユーザーIDまたはパスワードが誤っている場合は、ログイン画面にエラーメッセージを表示する。

ユーザーIDとパスワードはいずれも必須入力とする。

ユーザーIDは半角英数字のみ入力可能とし、最大20文字までとする。

パスワードは8文字以上、32文字以下とする。

認証に5回連続で失敗した場合は、アカウントを一時ロックする。

### 4.3 補足事項

エラーメッセージは、認証失敗理由を詳細に表示せず、共通メッセージを表示する。

アカウントロック解除は管理者操作で行う想定とする。

## 5. サンプル2：注文登録API

### 5.1 基本情報

| 項目 | 内容 |
|---|---|
| タイトル | 注文登録APIのテスト設計 |
| 対象種別 | `api` |
| テストレベル | `integration` |

### 5.2 仕様メモ

注文情報を受け取り、注文テーブルと注文明細テーブルに登録するAPI。

リクエストには、顧客ID、注文日、商品コード、数量、単価を含む。

注文には複数の注文明細を含めることができる。

顧客IDが存在しない場合はエラーとする。

商品コードが存在しない場合はエラーとする。

数量は1以上999以下とする。

注文登録時に在庫引当APIを呼び出す。

在庫引当APIが成功した場合のみ、注文情報をDBに登録する。

在庫引当APIが失敗した場合は、注文情報をDBに登録せず、エラーレスポンスを返す。

### 5.3 補足事項

注文登録と注文明細登録は同一トランザクションで処理する。

在庫引当失敗時はロールバックする。

レスポンスには注文ID、処理結果、メッセージを含める。

---

## 6. サンプル3：日次売上集計バッチ

### 6.1 基本情報

| 項目 | 内容 |
|---|---|
| タイトル | 日次売上集計バッチのテスト設計 |
| 対象種別 | `batch` |
| テストレベル | `system` |

### 6.2 仕様メモ

毎日深夜に実行される日次売上集計バッチ。

対象日の注文データを集計し、売上サマリテーブルを更新する。

集計単位は、売上日、店舗コード、商品カテゴリとする。

対象日の注文データが存在しない場合は、正常終了し、処理件数0件としてログ出力する。

注文データに不正な商品カテゴリが含まれる場合は、該当データをエラーとして扱い、エラーログに出力する。

エラーデータが存在する場合でも、正常データは集計対象とする。

バッチ終了時に、処理件数、正常件数、エラー件数をログ出力する。

異常終了した場合は、原因を取り除いたうえで再実行できる。

### 6.3 補足事項

再実行時は、対象日の売上サマリを削除してから再集計する。

バッチの実行結果は運用担当者がログで確認する。

## 7. サンプル4：顧客マスタ更新処理

### 7.1 基本情報

| 項目 | 内容 |
|---|---|
| タイトル | 顧客マスタ更新処理のテスト設計 |
| 対象種別 | `db` |
| テストレベル | `integration` |

### 7.2 仕様メモ

顧客マスタの登録・更新を行う処理。

顧客IDが未登録の場合は新規登録する。

顧客IDが既に登録済みの場合は、顧客名、住所、電話番号、メールアドレスを更新する。

顧客名は必須とする。

電話番号は半角数字とハイフンのみ許可する。

メールアドレスはメール形式であることを確認する。

更新時は、最終更新日時と更新者IDを更新する。

同一顧客IDに対して同時更新が発生した場合は、排他制御により後続更新をエラーとする。

処理中にDBエラーが発生した場合は、トランザクションをロールバックする。

### 7.3 補足事項

物理削除は行わず、退会済み顧客はステータスで管理する。

登録・更新の結果は処理ログに出力する。

---

## 8. サンプル5：在庫照会外部連携

### 8.1 基本情報

| 項目 | 内容 |
|---|---|
| タイトル | 在庫照会外部連携のテスト設計 |
| 対象種別 | `external` |
| テストレベル | `integration` |

### 8.2 仕様メモ

商品コードと倉庫コードを指定して、外部在庫管理システムへ在庫数を照会する処理。

外部在庫管理システムへHTTPリクエストを送信し、在庫数、引当可能数、応答日時を取得する。

外部システムが正常応答した場合は、取得した在庫情報を画面に表示する。

外部システムがエラー応答を返した場合は、在庫照会エラーとして扱う。

外部システムへの接続がタイムアウトした場合は、最大2回までリトライする。

リトライ後も失敗した場合は、エラーメッセージを表示し、連携失敗ログを出力する。

商品コードまたは倉庫コードが未入力の場合は、外部システムを呼び出さず入力エラーとする。

### 8.3 補足事項

タイムアウト時間は3秒とする。

外部連携エラー時は、画面上に詳細な技術情報を表示せず、利用者向けメッセージを表示する。

技術的な詳細はログに出力する。

## 9. Frontendでの利用方針

MVPでは、サンプル仕様をDBではなくFrontend側の定数として持ちます。

想定する保持項目は以下です。

| 項目 | 内容 |
|---|---|
| `id` | サンプル識別子 |
| `name` | サンプル名 |
| `title` | 入力フォームに設定するタイトル |
| `target_type` | 入力フォームに設定する対象種別 |
| `test_level` | 入力フォームに設定するテストレベル |
| `input_text` | 入力フォームに設定する仕様メモ |
| `supplemental_notes` | 入力フォームに設定する補足事項 |

サンプルを選択した場合、以下の入力フォーム項目に値を反映します。

- タイトル
- テスト対象種別
- テストレベル
- 仕様メモ
- 補足事項

```typescript
  supplementalNotes: string;

export const sampleSpecs: SampleSpec[] = [
  {
    id: "login-screen",
    name: "ログイン画面",
    title: "ログイン画面のテスト設計",
    targetType: "screen",
    testLevel: "acceptance",
    inputText: "ユーザーIDとパスワードを入力してログインする画面。...",
    supplementalNotes: "エラーメッセージは、認証失敗理由を詳細に表示せず、共通メッセージを表示する。"
  }
];
```

このドキュメントは設計メモであり、実装コードそのものは実装フェーズで作成します。

## 11. サンプル仕様の選定理由

各サンプル仕様を採用する理由は以下です。

| サンプル | 採用理由 |
|---|---|
| ログイン画面 | 画面入力、認証、エラー表示、権限観点を説明しやすい |
| 注文登録API | API、DB更新、外部連携、トランザクションを含めやすい |
| 日次売上集計バッチ | 業務系SE経験、バッチ、ログ、再実行観点を説明しやすい |
| 顧客マスタ更新処理 | DB更新、入力チェック、排他、整合性を説明しやすい |
| 在庫照会外部連携 | 外部連携、タイムアウト、リトライ、ログ観点を説明しやすい |

この5件により、業務系システム開発でよくある複数パターンをカバーできます。

## 12. セキュリティ・守秘義務上の注意

本ドキュメントのサンプル仕様はすべて架空データです。

以下の情報は含めません。

- 顧客名
- 個人情報
- 実在する会社名
- 実在するシステム名
- APIキー
- パスワード
- アクセストークン
- 本番環境情報
- 実際の設計書
- 実際のソースコード
- 実際の障害ログ
- 業務機密
- 契約上公開できない情報

今後サンプル仕様を追加する場合も、必ず架空データのみを使用します。

## 13. README・画面キャプチャでの利用方針

READMEや画面キャプチャでは、原則として以下のサンプルを利用します。

| 用途 | 推奨サンプル |
|---|---|
| メインのデモ画面 | 注文登録API |
| 業務系SE経験の説明 | 日次売上集計バッチ |
| 画面入力チェックの説明 | ログイン画面 |
| DB更新・排他の説明 | 顧客マスタ更新処理 |
| 外部連携・リトライの説明 | 在庫照会外部連携 |

最初のREADME用キャプチャでは、`注文登録API` を使う方針とします。

理由：

- API、DB、外部連携を含み、テスト観点が広がりやすい
- FastAPI / Backend経験との接続を説明しやすい
- 業務系システムらしさが出しやすい

---

## 14. MVPでは対応しないこと

Phase 1の初期MVPでは、以下は対応しません。

| 対象 | 理由 |
|---|---|
| サンプル仕様のDB管理 | 初期MVPではFrontend定数で十分なため |
| サンプル仕様の編集画面 | MVPの主目的から外れるため |
| サンプル仕様の追加画面 | 実装範囲が増えるため |
| サンプル仕様のインポート | ファイルアップロードが必要になり、範囲が広がるため |
| 実データを使ったサンプル | 守秘義務・セキュリティリスクがあるため |
| 複雑な業務フロー図 | MVPではテキスト仕様で十分なため |

## 16. 完了条件

サンプル仕様の完了条件は以下です。

- 画面/API/バッチ/DB更新/外部連携のサンプルが用意されている
- すべて架空データである
- 顧客情報、個人情報、APIキー、パスワード、業務機密を含まない
- Frontendのサンプル入力として利用できる粒度になっている
- READMEや画面キャプチャで使いやすい
- 面談時に業務系SE経験と接続して説明しやすい
