# セキュリティ・守秘義務方針

## 1. 目的

本ドキュメントは、「業務系SE向け AIテスト設計支援ツール」開発プロジェクトにおける、セキュリティ・守秘義務上の方針を整理するものです。

このプロジェクトでは、個人開発・ポートフォリオ用途であっても、本業の顧客情報や業務機密を使用しないことを最優先とします。

---

## 2. 入力禁止情報

アプリ、README、docs、サンプルデータ、テストデータ、GitHub Issues、commit内容には、以下を含めません。

- 顧客名
- 個人情報
- APIキー
- パスワード
- アクセストークン
- 本番環境情報
- 実際の設計書
- 実際のソースコード
- 実際の障害ログ
- 業務機密
- 契約上公開できない情報

---

## 3. アプリ上の注意書き

アプリ内には、以下の趣旨の注意書きを表示します。

```text
顧客名、個人情報、APIキー、パスワード、本番環境情報、業務機密を含む情報は入力しないでください。
必要に応じてマスキングしてから利用してください。
```

---

## 4. サンプルデータ方針

動作確認やデモでは、架空データのみを使用します。

使用してよい例：

- 架空のログイン画面
- 架空の注文登録API
- 架空の日次売上集計バッチ
- 架空の顧客マスタ更新処理
- 架空の在庫照会外部連携

使用してはいけない例：

- 本業で扱った画面仕様
- 実案件のAPI仕様
- 実際のDBテーブル定義
- 実際の障害ログ
- 実際の顧客名や利用者名

---

## 5. Git管理対象外

以下はGit管理対象に含めません。

```text
.env
.env.local
.venv/
node_modules/
.next/
__pycache__/
.pytest_cache/
*.db
*.sqlite
*.sqlite3
```

commit前には以下を確認します。

```powershell
git status --short
```

必要に応じて、以下も確認します。

```powershell
git ls-files | Select-String -Pattern "\.env$|\.env\.local$|\.db$|\.sqlite$|\.sqlite3$|node_modules|\.next|\.venv"
```

何も表示されなければ、対象ファイルはGit管理対象に含まれていません。

---

## 6. 環境変数方針

環境変数の実値はGitHubに載せません。

- 実値は `.env` または `.env.local` に置く
- `.env` と `.env.local` はGit管理対象外にする
- 設定例のみ `.env.example` としてGit管理する
- APIキー、パスワード、アクセストークンはREADMEやdocsにも書かない

Backendのデフォルトは `APP_LLM_PROVIDER=mock` のため、外部LLM APIキーは不要です。
`APP_LLM_PROVIDER=openai` を指定する場合のみ `OPENAI_API_KEY` が必要です。
`OPENAI_API_KEY`、`.env`、`.env.local` の実値はGitHubに載せません。

---

## 7. SQLite DBファイルの扱い

ローカルMVPでは、生成履歴をSQLite DBに保存します。

開発用DBファイルはGit管理対象外です。

```text
backend/app.db
```

Backend再起動後も履歴は保持されます。履歴を初期化したい場合は、Backend停止後に開発用SQLite DBファイルを手動削除します。

---

## 8. 外部LLM利用時の注意

Phase 1のローカルMVPでは、OpenAI API連携は任意機能です。
未指定時はMock生成を使用し、OpenAI APIは呼び出しません。

OpenAI APIを利用する場合は、以下を必ず確認します。

- APIキーをGitHubへ上げない
- 入力内容に機密情報を含めない
- 実案件情報、顧客情報、個人情報、業務機密を入力しない
- 利用回数や料金上限を設定する
- `OPENAI_MODEL` はallowlist内で切り替える
- モデル変更時はOpenAI公式の料金・品質・速度・利用制限を確認する
- エラー時やタイムアウト時の扱いを決める
- 必要に応じてマスキング処理を入れる
- CIではOpenAI APIを呼び出さない

Amazon Bedrock連携は未実装です。

---

## 9. AWS利用時の注意

AWSはPhase 2以降で扱います。

AWS利用時は、以下を徹底します。

- AWS Pricing Calculatorで料金見積もりを確認する
- AWS Budgetsで予算アラートを設定する
- 常時稼働リソースを避ける
- 検証後に不要リソースを削除する
- CloudWatch Logsの保持期間を短くする
- デプロイ手順だけでなく削除手順もdocsに残す

NAT Gateway、ALB、RDS、ECS Fargateはコストが発生しやすいため、初期段階では原則避けます。

---

## 10. セキュリティ違反が疑われる場合

ユーザーが意図せず機密情報を入力・共有しようとしている場合は、通常の実装支援よりもセキュリティ上の安全性を優先します。

対応方針は以下です。

1. セキュリティ上の問題点を明確にする
2. その情報を使わないよう案内する
3. マスキング方法または架空データへの置き換え方法を提示する
4. 必要に応じて安全なサンプルデータを作成する
5. リスクが解消されるまで、その情報を前提にした実装・要約・変換・設計支援は行わない
