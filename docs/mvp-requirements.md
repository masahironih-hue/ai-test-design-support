# MVP要件定義

## 1. 目的

本ドキュメントは、「業務系SE向け AIテスト設計支援ツール」における Phase 1：ローカルMVP の要件を整理するものです。

Phase 1 では、まずローカル環境で以下の流れが動く状態を目指します。

```text
仕様入力
↓
LLM Mockによるテスト観点生成
↓
生成結果表示
↓
Markdownコピー
↓
履歴保存
↓
履歴一覧・履歴詳細確認
```

このMVPの主目的は、AIの高度な生成精度を追求することではなく、業務系SEの実務経験を活かしたテスト設計支援ツールとして、Webアプリケーションの一連の流れを説明可能な形で実装することです。

重視する観点は以下です。

- 個人開発として継続しやすいこと
- GitHubポートフォリオとして説明しやすいこと
- Python / FastAPI 案件への接続材料になること
- React / Next.js / TypeScript 案件への接続材料になること
- AI / LLM 案件への接続材料になること
- 業務系SE経験と接続できること
- 守秘義務・セキュリティ上のリスクを避けること

## 更新メモ

本ドキュメントは、ローカルMVP総合動作確認後の実装状態に合わせて、APIパス、入力項目名、`target_type` / `test_level` のコード値を更新しています。

- 生成API: `POST /test-designs/generate`
- 履歴一覧API: `GET /test-designs/histories`
- 履歴詳細API: `GET /test-designs/histories/{history_id}`
- 入力項目: `title`、`target_type`、`test_level`、`spec_text`、`supplement`
- `target_type`: `screen`、`api`、`batch`、`db`、`external`
- `test_level`: `unit`、`integration`、`system`

## 2. MVPで作る機能

Phase 1 のMVPでは、以下の機能を作成します。

| 機能 | 概要 |
|---|---|
| 仕様入力フォーム | テスト対象となる仕様メモを入力する |
| テスト対象種別の選択 | 画面、API、バッチ、DB更新、外部連携などを選択する |
| テストレベルの選択 | 単体テスト、結合テスト、システムテスト、受入テストなどを選択する |
| LLM Mockによる生成 | OpenAI API等は使わず、固定ロジックでテスト観点を生成する |
| 生成結果表示 | テスト観点、テストケース案、追加確認事項を画面表示する |
| Markdown出力 | 生成結果をMarkdown形式で出力する |
| Markdownコピー | 生成されたMarkdownをクリップボードへコピーできるようにする |
| 履歴保存 | 入力内容と生成結果をSQLiteに保存する |
| 履歴一覧 | 過去の生成履歴を一覧表示する |
| 履歴詳細 | 過去の入力内容・生成結果・Markdownを確認する |
| セキュリティ注意書き表示 | 機密情報を入力しないよう注意喚起を表示する |
| サンプル仕様 | 架空データによるサンプル仕様を用意する |

---

## 3. MVPで作らない機能

Phase 1 のMVPでは、以下の機能は作り込みません。

| 対象 | 理由 |
|---|---|
| OpenAI API連携 | まずはMockで画面/API/DBの流れを安定させるため |
| Amazon Bedrock連携 | Phase 4以降の拡張候補とするため |
| AWSデプロイ | Phase 2で扱うため |
| Docker必須化 | 初期の環境構築負荷を下げるため |
| 認証 | ローカルMVPでは不要なため |
| 課金 | MVPの目的から外れるため |
| マルチテナント | 初期段階では過剰設計となるため |
| ファイルアップロード | セキュリティリスクと実装範囲が増えるため |
| Excel出力 | Markdown出力後の拡張で十分なため |
| RAG | 初期MVPには不要なため |
| プロンプトテンプレート管理 | まずは固定Mockで十分なため |
| 生成結果の編集 | 初期は表示とコピーに限定するため |
| 履歴検索・フィルタ | 履歴件数が少ない前提では不要なため |
| 履歴削除 | MVP後の改善候補とするため |
| 高度なUI装飾 | 業務アプリとしての分かりやすさを優先するため |

## 4. 画面構成

MVPでは、以下の3画面を作成します。

### 4.1 テスト設計生成画面

パス案：

```text
/
```

役割：

- 仕様メモを入力する
- テスト対象種別を選択する
- テストレベルを選択する
- LLM Mockによる生成を実行する
- 生成結果を表示する
- Markdownをコピーする
- セキュリティ注意書きを表示する

入力項目：

| 項目 | 型 | 必須 | 備考 |
|---|---|---:|---|
| タイトル | text | 任意 | 未入力時は自動生成 |
| テスト対象種別 | select | 必須 | 画面 / API / バッチ / DB更新 / 外部連携 / その他 |
| テストレベル | select | 必須 | 単体 / 結合 / システム / 受入 |
| 仕様メモ | textarea | 必須 | メイン入力 |
| 補足事項 | textarea | 任意 | 前提条件、制約、追加観点など |

---

### 4.2 履歴一覧画面

パス案：

```text
/history
```

役割：

- 過去の生成履歴を一覧表示する
- 作成日時、タイトル、対象種別、テストレベルを表示する
- 履歴詳細画面へ遷移する

表示項目：

| 項目 |
|---|
| 作成日時 |
| タイトル |
| テスト対象種別 |
| テストレベル |
| 詳細リンク |

MVPでは、検索、フィルタ、削除、編集は作成しません。

---

### 4.3 履歴詳細画面

パス案：

```text
/history/[id]
```

役割：

- 過去の入力内容を表示する
- 過去の生成結果を表示する
- Markdownを表示する
- Markdownをコピーする

MVPでは、履歴の編集機能は作成しません。

## 5. API構成

MVPで作成するAPIは以下です。

| メソッド | パス | 用途 |
|---|---|---|
| GET | `/health` | ヘルスチェック |
| POST | `/test-designs/generate` | テスト設計生成 + 履歴保存 |
| GET | `/test-designs/histories` | 履歴一覧取得 |
| GET | `/test-designs/histories/{history_id}` | 履歴詳細取得 |

---

用途：

- FastAPIの起動確認
- ローカル環境での疎通確認

## 6. DB構成

MVPでは、履歴保存用のテーブルを1つ作成します。

### 6.1 テーブル名

```text
test_design_histories
```

### 6.2 カラム

| カラム | 型 | 必須 | 内容 |
|---|---|---:|---|
| id | integer | 必須 | 主キー |
| title | string | 必須 | タイトル |
| target_type | string | 必須 | テスト対象種別 |
| test_level | string | 必須 | テストレベル |
| spec_text | text | 必須 | 入力された仕様メモ |
| supplement | text | 任意 | 補足事項 |
| result_json | text | 必須 | 生成結果JSON文字列 |
| result_markdown | text | 必須 | Markdown文字列 |
| llm_provider | string | 必須 | 初期値は `mock` |
| llm_model | string | 必須 | 初期値は `mock-v1` |
| created_at | datetime | 必須 | 作成日時 |
| updated_at | datetime | 必須 | 更新日時 |

MVPでは、テスト観点やテストケースを別テーブルに分けず、1つの履歴データとして保存します。

理由：

- 実装を単純にするため
- 初期開発の負荷を下げるため
- 入力と出力の対応関係を保持しやすくするため
- 後から正規化・検索対応を検討できるため

## 7. LLM Mock仕様

### 7.1 方針

MVPでは、外部LLM APIを呼び出さず、Mockサービスでテスト観点を生成します。

Mockの方針は以下です。

| 方針 | 内容 |
|---|---|
| 決定論的であること | 同じ入力に対して同じ結果を返す |
| 対象種別で分岐すること | 画面/API/バッチ/DB更新/外部連携で一部観点を変える |
| JSON構造を固定すること | フロントエンドで表示しやすくする |
| 高度な自然言語処理はしないこと | MVPでは実装範囲を広げすぎない |
| 将来差し替え可能にすること | OpenAI APIやBedrock連携に拡張しやすくする |

---

### 7.2 入力

Mockサービスへの入力例：

```json
{
  "title": "注文登録APIのテスト設計",
  "target_type": "api",
  "test_level": "integration",
  "spec_text": "注文情報を受け取り、注文テーブルと注文明細テーブルに登録するAPI。",
  "supplement": "在庫引当処理は外部API連携で行う。"
}
```

---

### 7.3 出力

Mockサービスは、以下のような固定構造のJSONを返します。

```json
{
  "summary": "注文登録APIに対するテスト観点とテストケース案です。",
  "viewpoints": [
    {
      "category": "正常系",
      "items": [
        "必須項目をすべて指定した場合に処理が成功すること",
        "複数明細を含むデータが正しく処理されること"

    },
    {
      "category": "異常系",
      "items": [
        "必須項目が未入力の場合にエラーとなること",

      ]
    },
    {
      "category": "DB更新確認",
      "items": [
        "対象テーブルにデータが登録されること",
        "関連テーブルとの整合性が保たれること"

  "test_cases": [
    {
      "case_no": "TC-001",
      "title": "正常なデータ登録",

        "正常な入力データを送信する",
        "レスポンス内容を確認する",

      "expected_result": "処理が成功し、期待するデータが登録される",

  ]
```

## 8. 生成結果JSON形式

MVPで扱う生成結果JSONは、以下の構造を基本とします。

```json
{
  "summary": "生成結果の概要",
  "viewpoints": [
    {
      "category": "正常系",
      "items": [
        "観点1",
        "観点2"
      ]
    }
  ],
  "test_cases": [
    {
      "case_no": "TC-001",
      "title": "テストケース名",
      "precondition": "前提条件",
      "steps": [
        "手順1",
        "手順2"
      ],
      "expected_result": "期待結果",
      "priority": "High"
    }
  ],
  "additional_questions": [
    "追加確認事項1",
    "追加確認事項2"
  ]
}
```

この形式にする理由は以下です。

- 画面表示しやすい
- Markdown変換しやすい
- DBに保存しやすい
- 将来のOpenAI APIレスポンス整形に使いやすい
- 将来のExcel出力に拡張しやすい

## 9. Markdown出力形式

生成結果はMarkdown形式でも出力します。

出力形式は以下を基本とします。

```markdown
# テスト設計結果

## 入力情報

- タイトル: 注文登録APIのテスト設計
- テスト対象種別: API
- テストレベル: 結合テスト

## 概要

注文登録APIに対するテスト観点とテストケース案です。

## テスト観点

### 正常系

- 必須項目をすべて指定した場合に処理が成功すること
- 複数明細を含むデータが正しく処理されること

### 異常系

- 必須項目が未入力の場合にエラーとなること
- 存在しないコード値が指定された場合にエラーとなること

### DB更新確認

- 対象テーブルにデータが登録されること
- 関連テーブルとの整合性が保たれること

## テストケース案

| No | タイトル | 前提条件 | 手順 | 期待結果 | 優先度 |
|---|---|---|---|---|---|
| TC-001 | 正常なデータ登録 | 必要なマスタデータが登録済みである | 正常な入力データを送信する | 処理が成功し、期待するデータが登録される | High |

## 追加確認事項

- エラー発生時のメッセージ仕様を確認する必要があります。
- 外部連携失敗時のリカバリ方針を確認する必要があります。

## 注意事項

この出力はAIによるテスト設計支援結果です。
最終的なテスト観点・期待結果は、実際の仕様書、設計書、業務ルールに基づいて確認してください。
```

## 10. 履歴保存方針

履歴保存の方針は以下です。

| 項目 | 方針 |
|---|---|
| 保存タイミング | 生成成功時に自動保存する |
| 保存対象 | 入力内容、生成JSON、Markdown、Mock情報、作成日時 |
| 編集 | MVPでは作らない |
| 削除 | MVPでは作らない |
| 検索 | MVPでは作らない |
| 並び順 | 作成日時の降順 |
| タイトル未入力時 | `テスト設計 yyyy-MM-dd HH:mm` の形式で自動生成する |

履歴保存の目的は以下です。

- FastAPIでDBアクセスしていることを示す
- SQLiteで永続化していることを示す
- フロントエンドから一覧・詳細を取得していることを示す
- 業務アプリでよくある一覧・詳細構成を実装する

## 11. サンプル仕様方針

サンプル仕様は、すべて架空データで作成します。

本業の顧客情報、実コード、実ログ、実設計書、個人情報、APIキー、パスワード、業務機密は使用しません。

MVPで用意するサンプル候補は以下です。

| サンプル | 対象種別 | 狙い |
|---|---|---|
| ログイン画面 | 画面 | 入力チェック、認証、エラー表示を説明しやすい |
| 注文登録API | API | 正常系、異常系、DB更新、外部連携を含めやすい |
| 日次売上集計バッチ | バッチ | 業務系SE経験と接続しやすい |
| 顧客マスタ更新処理 | DB更新 | 登録、更新、排他、整合性確認を説明しやすい |
| 在庫照会外部連携 | 外部連携 | タイムアウト、リトライ、ログ確認を説明しやすい |

MVPでは、サンプル仕様をDB管理せず、フロントエンド側のサンプル入力として持たせる方針とします。

## 12. セキュリティ注意書き

生成画面には、仕様入力欄の近くに以下の注意書きを常時表示します。

```text
注意：
顧客名、個人情報、APIキー、パスワード、本番環境情報、業務機密を含む情報は入力しないでください。
必要に応じて、架空データへの置き換えやマスキングを行ってから利用してください。
```

表示・記載対象は以下です。

| 場所 | 方針 |
|---|---|
| 仕様入力フォーム上部 | 必須 |
| Markdown出力内の注意事項 | 推奨 |
| README | 必須 |
| docs/security-policy.md | 必要に応じて追記 |

この注意書きは、業務システム開発における守秘義務・情報管理を意識していることを示すためにも重要です。

Phase 1では、以下の順序で実装します。

### Step 1: MVP仕様をdocs化する

- 実装スコープを固定する
- Codexへ実装依頼しやすくする

### Step 2: Backend最小構成を作成する
